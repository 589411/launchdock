> 建立:2026-07-05。每條禁止規則都標注它防的實際錯誤(E 編號),沒有實例的規則不准加。

## 1. 受眾與目的

讀者是**台灣非技術背景的 SME 決策者與學習者**:上班族、店家、老師,聽過 ChatGPT 但不懂 API。
每篇文章要讓讀者帶走**一件明天就能動手做的事**。
商業目的是建立教學可信度、導向企業培訓與課程——**不是衝流量**。因此:寧可少而準,不可多而水。

## 2. 品牌聲音(可操作規則)

角色定義正本在 `.github/agents/blue-duck.agent.md`(比喻資料庫、鴨編的話格式)。摘要成規則:

1. **用「你」對讀者說話,口語但專業。**
   - 好:「用了三個月之後,它有沒有變得更了解你?」(ai-agent-memory-guide)
   - 壞:「本文將探討 AI Agent 記憶機制之架構」
2. **技術名詞首次出現必配生活化比喻或一句白話。** 比喻優先查 blue-duck 的比喻資料庫,保持全站一致(Token=電話費、API Key=門禁卡)。
3. **開頭 3 行內給讀者價值**:一個問題、一個場景、或一個反直覺的事實。不從定義開場。
4. **結尾必有行動指引**:讀完後做什麼(對應 lab 條目的 `tomorrow` 精神)。
5. **直接、務實、不弱智化**:可以吐槽、可以說「這個坑我踩過」,不可以裝可愛湊字數。

## 3. 文章類型(對應 frontmatter `contentType`)

| 類型 | contentType | 必要區塊 | 長度 |
|------|-------------|---------|------|
| 教學文 | tutorial | 前置需求 → 編號步驟(配 @img)→ 常見錯誤 🚨 → 下一步 | 10–20 min |
| 觀念文 | guide | 開場問題 → 概念拆解(配概念圖)→ 實務判斷 → 行動指引 | 8–15 min |
| 速查文 | reference | 表格/清單為主,少敘事 | 3–8 min |
| 踩坑文 | troubleshoot | 症狀 → 根因 → 解法 → 如何預防(見 `templates/pitfall-to-article.md`) | 5–12 min |
| 碎碎念 | guide + scene: 鴨編的碎碎念 | 觀點文,可主觀,但事實仍須查證 | 5–15 min |

## 4. 硬規則(可機械檢查,違反不得發佈)

- H1. `npm run build` 必須通過(schema 驗證含在內)。
- H2. 內文不得殘留佔位符:`REPLACE`、`TODO`、`FIXME`、`XXX`。【防 E2、E4】
- H3. 圖片規則:**概念圖/示意圖/流程圖一律 AI 生成 SVG,不准留 @img 欠條**;只有真實 UI 操作截圖才允許 @img 待補。【防 E1】
- H4. 新增/更新文章必跑 `npm run registry`;新概念必登錄 `concepts.yaml`。
- H5. 中英雙版同時產出,slug 相同(規則見 llm-article-prompt.md)。
- H6. 任何插圖進 `public/images/` 後必跑機敏掃描(CLAUDE.md 安全規則)。
- H7. 引用外部工具的價格、版本、功能,必須當天查證,並更新 frontmatter `verifiedAt`。【防 E5】

## 5. 判斷指引(給模型權衡,優先序由上而下)

1. 讀者能不能照著做 > 內容完整性(砍掉做不到的段落)
2. 事實正確 > 比喻生動(比喻扭曲事實就換比喻)
3. 一篇講透一件事 > 一篇覆蓋十件事(多主題拆文,用 prerequisites 串)
4. 品牌聲音 > SEO 用語

## 6. 禁止清單(每條對應實際錯誤)

- 不准以「發佈後再補」為由留下佔位符或空連結。【E2:lab 曾有 3 筆 REPLACE 死連結以 active 上線】
- 不准生成需要人工截圖的概念示意 @img。【E1:缺圖債曾累積 91 張、兩週零進展】
- 不准複製本檔內容到其他檔案,只准引用路徑。【E3:loop 制度文件曾斷鏈腐化(dev-harness 檔案被 archive 但三處還在引用)】
- 草稿(`_drafts/`)轉正式前必須通過 H1–H7 全部檢查。【E4:曾有含多個 TODO 的草稿】

## 7. 「完成」的定義(產線見 templates/)

機械檢查全過(H1–H7)+ 角色審查完成(工程師D 查證、新手B 讀者審)+ **Joseph 人工 gate**。
任何文章不經人工 gate 不得 commit 發佈。AI 不得代替 Joseph 按下發佈。

## 8. 角色分工(正本在 `.github/agents/`)

| 角色 | 產線位置 | 職責 |
|------|---------|------|
| 工程師D | 成文**前** | 查證觀察/宣稱:web 搜尋、技術正確性、時效;輸出查證備忘(確認/修正/不確定三類) |
| 藍鴨 | 成文 | 依本檔 + llm-article-prompt.md 寫中英雙版 |
| 新手B | 成文**後** | 以零基礎視角挑「看不懂的地方」,產出 stuckOptions |
| 管理者C | 不在產線 | 社群互動、FAQ,與文章生成無關 |
