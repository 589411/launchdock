# vibe coding 試營運 — demo 任務課程腳本（候選）

> 配套：`docs/ROADMAP.md`（限時能力包實驗）。目的：挑「普通模型 ＋ 藍鴨能力包」能**穩定完成**的 demo 任務，驗 H3。
> 來源：`~/github/launchdock-lab/data/projects.yaml`（lab demo 庫）＋ `~/github/line-booking-course`（+ 答案版 `line-booking-system`，專為教學設計的配對 repo）。
> ⚠️ **誠實前提**：這是照 metadata／README 挑「符合保證會贏條件」的候選，**尚未用普通模型實跑驗證**。上場前必做 dry-run（見文末）——那一步才是真正驗 H3。

---

## 挑選標準（為什麼是這三個）
一個 demo 任務能不能在直播現場「保證會贏」，看四件事：
1. **依賴少**：不需金鑰、不需複雜部署授權、不吃會變動的外部資料。
2. **輸出可見**：學員當場看得到成果，有成就感。
3. **一節課做得完**：範圍框得住。
4. **能凸顯能力包價值**：正好避開普通模型自己做會踩的坑。

依這四點排序後，選 **核心 A（最穩暖身）＋ 核心 B★（首選 SME 主菜）＋ 核心 B-備選（通用題材）＋ stretch（需先驗）**。

---

## 核心 A（最穩，文件輸出）— 用 AI 寫你的第一份規格書
- **lab 專案**：`sdd-learning-guide`（L2、GitHub Pages、規格驅動開發 SDD 教材）
- **目標產出**：一份針對學員自己想做的小工具的**軟體規格書**（需求→規格）。
- **為什麼保證會贏**：輸出是文件、不是會跑的系統，**近乎零失敗**。而且「先寫規格再動手」正是能力包 / `dev-harness` 的方法核心——這一關贏在方法，不贏在運氣。
- **能力包在這幫什麼**：把 SDD 的提問框架、規格範本、常見漏項 checklist 餵給普通模型，讓它問對問題、不跳步。普通模型自己寫規格常常太空泛、漏邊界條件——能力包補的就是這個。
- **現場流程**：① 學員一句話說想做什麼 → ② 能力包引導模型反問澄清 → ③ 產出結構化規格 → ④ 當場檢查「照這份規格，工程師/AI 做得出來嗎」。
- **失敗點 & fallback**：幾乎無。真的卡住 → 換更小的題目（例：一個計算機、一張報名表）。
- **對應 module**：M02。

## 核心 B★（首選 SME 主菜）— LINE 會員預約系統：挑「一課」現場做出來
- **來源 repo**：`589411/line-booking-course`（學員乾淨起點，L0–L8 閘門課＋`安裝包/` 參考真理）｜答案版 `589411/line-booking-system`（完整可跑，現場翻車時對照）。技術：LIFF＋Firebase Firestore＋Cloudflare Worker＋PWA。
- **為什麼升為首選**：它**本身就是課程骨架**——L0–L8 每課「框死範圍＋關卡＋驗收＋STOP」，正好命中「範圍框得住」；`安裝包` 讓學員接現成核心而非從零寫；而且**自帶答案版 repo＝現成 fallback**（別的 demo 都得另外準備）。主題（工作室賣堂數/排課/點名）對台灣 SME 極有感，wow 最高。
- **第一場只做一課，別做全套**：首選 **L3「LIFF 認人與查堂數」**——學員 LINE 一打開，系統就認得他、秀出剩餘堂數。「它認得我」的瞬間 aha 值爆表，但範圍可控（前端＋LIFF，後端資料可預先塞）。
- **能力包在這幫什麼**：把 LIFF 初始化、Firebase 讀取、常見授權坑收成骨架＋checklist，讓普通模型不卡在 console 設定。
- **⚠️ 上場前預先設定（把最會踩雷的外部設定先做掉，這步不能省）**：
  1. 預建 **LINE Developers channel ＋ LIFF ID**（共用 demo 用）。
  2. 預建 **共用 demo Firebase 專案**、套好 Firestore 規則、塞好假會員/堂數資料。
  3. 確認 **GitHub Pages** 可部署前端。
  → 讓學員現場 **vibe code 應用邏輯**，不是跟主控台設定搏鬥（同 gas-ordering 的教訓）。
- **失敗點 & fallback**：LIFF/Firebase 設定錯 → 對照 `line-booking-system` 答案版；範圍失控 → 只做「LINE 開啟→認得我→顯示堂數」最小閉環。
- **對應 module**：M04（LINE）＋ M02（部署）。

## 核心 B-備選（可見的建置，自足前端、零外部設定）— 做一個單頁互動網站並上線
- **lab 專案**：`bible-atlas`（L3、GitHub Pages、純前端像素風互動地圖/時間軸）
- **目標產出**：一個**單檔前端互動網頁**（時間軸、地圖、小工具、資料視覺化擇一），部署到 GitHub Pages。主題換成學員自己在意的（不必是聖經）。
- **為什麼保證會贏**：**純 client-side、無金鑰、無後端**，錯了就改、當場可迭代，失敗可回復。單檔互動網頁是 vibe coding 最甜的區間，普通模型產出率高。成果一上線就有 URL 可分享＝高成就感。
- **能力包在這幫什麼**：給「單檔 HTML＋JS」的骨架範本、避開常見坑（外部 CDN 抓不到、路徑錯、GitHub Pages 設定），讓普通模型少走冤路。站上已有〈部署到 GitHub Pages〉可直接接。
- **現場流程**：① 選型（時間軸/地圖/工具）→ ② 能力包給骨架 → ③ 填內容、逐步加互動 → ④ 推 GitHub Pages 上線 → ⑤ 拿到 URL。
- **失敗點 & fallback**：範圍失控（想加太多）→ 先做「能動的最小版」再加。GitHub Pages 沒設好 → 站上文章排錯。
- **對應 module**：M02。

## Stretch（高關聯、高 wow，但**務必先 dry-run**）— 小店訂單系統
- **lab 專案**：`gas-ordering`（L3、GAS＋Google Sheets、同網址 `?page=kitchen` 切換前後台）
- **目標產出**：一個**能收單、寫進 Google Sheet 的訂餐系統**，前台給顧客、後台給廚房。
- **為什麼放 stretch 不放核心**：對店家客群**最有感、wow 最大**，也**最能凸顯能力包價值**（GAS 部署授權、doGet 分流、Sheet 權限是普通模型最會踩的坑，能力包正好收掉）。**但**同樣因為 GAS 部署/授權雷多，**現場失敗風險最高**。
- **上場條件**：你**親自用普通模型＋能力包跑通一次**、確認每個部署步驟穩定，才放進正式場；否則只當「示範觀摩」不讓學員自己跑。
- **對應 module**：M03、M04。

## 建議組合
- **第一場試營運（推薦）**：核心 A（規格書，暖身、穩贏）→ **核心 B★ 的 L3 一課**（LINE 認人＋查堂數，主菜、SME 高 wow、有答案版兜底）。前提是**外部設定已預先做掉**。
- **若不想扛 LINE/Firebase 預設定**：改用 **核心 B-備選**（單頁網站上線，零外部設定、可回復）當主菜——最保險的第一場。
- **gas-ordering**：關聯性已被核心 B★ 取代，除非有特定店家客群、且你 dry-run 過，否則可略。

## 上場前必做：dry-run 驗 H3（這步不能跳）
用你打算讓學員用的**那個普通/便宜模型 ＋ 能力包**，把要上的那兩個任務**各跑一次**，全程照現場流程走一遍，記錄：
1. **完成率**：有沒有做出可用成果？（這就是 roadmap 的第三個數字）
2. **卡點**：模型在哪裡卡住？能力包有沒有把它接住？沒接住的坑 → 補進能力包（`589411/memory`）。
3. **時間**：一節課做得完嗎？做不完 → 再縮範圍。

> 核心 B★（LINE 課程）的 dry-run 特別關鍵：它是所有候選裡外部依賴最重的，也最能回答「安裝包＋一課能不能讓普通模型獨立完成」——這比通用 demo 更真實地驗 H3，同時測出你的 `line-booking-course` 是否已經是一個夠好的「能力包」。

dry-run 過了，「保證會贏」才成立；沒過就先修任務或能力包，別急著開賣（＝roadmap 翻案條件第一條）。
