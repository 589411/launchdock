# Launchdock 內容 Harness 建置提示詞

> 使用方式:在 launchdock 內容 repo 根目錄啟動 Claude Code(Fable 5),貼上以下全文。
> 【】內的欄位請先填好再貼。

## 角色與總目標

你是我的 harness engineer。這次 session 的任務**不是寫任何一篇文章**,而是建立一套制度,讓之後每天執行的弱模型 session 能穩定產出符合我標準的文章,並且每次犯錯都能把教訓寫回制度。

我的背景:我是藍鴨 Launchdock(launchdock.app)的經營者,受眾是台灣非技術 SME 決策者。內容的商業目的是建立教學可信度、導向企業培訓與課程,**不是衝流量或量產**。優化目標是:單篇產出成本下降,品質不降。

現有基礎(請以實際掃描結果為準,不要假設):
- 半自動化的截圖轉文章 workflow
- Claude Code skills:lab-publish、lab-maintain
- GitHub Pages + GitHub Actions + Cloudflare 部署
- 【其他你想讓它知道的現況,例如文章目錄位置】

## 全程守則(違反即停)

1. 不寫任何文章。只建制度。
2. 每個 Phase 結束必須停下來,給我看產出、等我確認才能進下一個 Phase。
3. 每一條你寫進制度的規則,必須標注它防的是 Phase 0 診斷出的哪一個實際錯誤。沒有對應實例的規則不准寫——想像出來的規則是指令檔肥大的來源。
4. 任何「完成」的宣稱都必須附上驗證證據(檔案路徑、檢查腳本輸出、回讀結果)。
5. 制度文件用繁體中文寫,因為之後讀它的是我,不只是模型。

## Phase 1|制度正本 `harness/EDITORIAL.md`

這是唯一正本,其他檔案只准引用它,不准複製它的內容(防漂移)。內容:

1. **受眾與目的**:一段話定義讀者是誰、讀完要帶走什麼、這篇內容如何服務藍鴨的商業目的。
2. **品牌聲音**:3–5 條可操作的語氣規則,每條附一組「好/壞」對照例句(從我的既有文章裡挑,不要自己編)。
3. **文章類型與結構範本**:教學文、案例文、觀點文、工具評測——各自的必要區塊與長度區間。
4. **硬規則 vs 判斷指引分開列**:硬規則是可機械檢查的;判斷指引是給模型權衡用的,並說明權衡的優先序。
5. **禁止清單**:直接從 Phase 0 的失敗模式反推,每條標注來源錯誤編號。

## Phase 2|品質 Rubric 與驗收定義

1. `harness/checks/` 下寫機械檢查腳本(可跑在 CI 或本地):連結有效性、圖片存在、meta 完整、字數區間、佔位符偵測、【你要加的其他檢查】。
2. `harness/JUDGE.md`:LLM 評審的固定 rubric prompt。評分維度限 4–6 個,每個維度必須定義 1 分和 5 分的具體樣貌,輸出格式為結構化 JSON + 每個維度一條具體修改建議。註明:judge 分數只用於**回歸偵測**(這批比上批差了沒),不作為發佈依據。
3. 在 EDITORIAL.md 明確寫下「完成」的定義:機械檢查全過 + judge 無紅燈維度 + **人工 gate(我點頭)**。任何文章不經人工 gate 不得發佈。

**停。跑一次:拿 Phase 0 抽樣過的其中兩篇舊文章,實際執行機械檢查 + judge,把結果給我看,證明 rubric 能抓到 Phase 0 診斷過的問題。抓不到就修 rubric,不准進下一步。**

## Phase 3|任務範本(弱模型日常用)

建立 `harness/templates/article-task.md`,這是之後每次生成文章的標準開場,包含:

1. 輸入格式:素材來源(截圖/主題/參考連結)怎麼給。
2. 必讀:指向 EDITORIAL.md 的薄索引(只列「這類任務要看哪幾節」,不複製內容)。
3. 工作步驟:草稿 → 自跑機械檢查 → 自跑 judge → 依建議修一輪 → 回報。
4. 回報格式:檢查結果原文照貼、judge JSON 照貼、自我標注「我不確定的地方」清單。
5. 強制條款:未附檢查證據的「完成」視為未完成;同一問題修兩輪還不過,停下來問人,不准疊 patch。

## Phase 4|Loop 機制

1. `harness/LESSONS.md` 踩坑帳本,固定格式:日期|症狀|根因|制度修正(改了 EDITORIAL/JUDGE/checks 的哪一條)。**只記帳不改制度的教訓視為沒學到。**
2. `harness/MAINTENANCE.md` 維護協議:
   - 我每次退件或人工修改,觸發一筆 LESSONS 記錄(這是主訊號)。
   - 每【10】篇做一次回顧 session:讀 LESSONS,檢查是否有重複症狀 → 有就升級為制度修正;同時檢查 EDITORIAL.md 是否有從未被引用的規則 → 有就刪(制度也要減肥)。
   - Cloudflare Analytics 與 judge 分數趨勢為輔助訊號,只用於季度回顧,不驅動單篇決策。
3. 定義升級條件:什麼情況弱模型該換強模型(例如:觀點文的論證架構、新文章類型的首篇)。

## Phase 5|對抗審查與回讀驗證

1. 開一個子任務扮演反方審查者,只給它看你建立的所有 harness 檔案,要求它找出:規則互相衝突處、弱模型可能鑽的漏洞、會導致指令檔膨脹的設計、我(人類)每篇要花超過 5 分鐘的環節。
2. 依審查結果修正。
3. 回讀驗證:列出所有建立的檔案路徑,逐一確認存在、互相引用的路徑正確、薄索引沒有複製正本內容。
4. 產出 `harness/README.md`:一頁使用說明——日常怎麼啟動一次文章生成、退件時怎麼記 LESSONS、回顧 session 怎麼跑。

**最終交付:AUDIT、EDITORIAL、JUDGE、checks/、templates/、LESSONS、MAINTENANCE、README,外加你對這套制度最大弱點的一段誠實評估。**
